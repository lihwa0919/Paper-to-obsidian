# Obsidian Note Template

## File format

```markdown
1. 해결하고자 한 핵심 문제 
	[핵심 문제 설명 — 선행 연구의 한계, 본 논문이 풀려는 문제를 구체적으로 기술]

	세부적으로는 두 가지를 함께 풀려고 합니다.
	- [한계 1] → [해결 방향]
	- [한계 2] → [해결 방향]

2. 사용한 구체적 방법론 (툴/알고리즘)
	[방법론 설명 — 사용한 모델/알고리즘/도구, 데이터셋, 파이프라인 설명]
	 핵심 성과: [주요 성능 지표 및 수치]

3. 서론에 인용하기 좋은 핵심 문장 한 줄
	[인용 가능한 한국어 문장. 예: "최근 연구에서는 X 방법이 Y 문제를 해결하기 위해 Z를 활용함을 보였으며[ ], ..."]
```

## Real examples from the vault

### Example 1

```markdown
1. 해결하고자 한 핵심 문제 
	시공 진척을 충분히 자주·정확하게 파악하지 못해 발생하는 일정·예산 관리 실패를, 이미 현장에서 수집 중인 시각 데이터(이미지·레이저 스캔)로 자동 측정하는 것.

	세부적으로는 두 가지를 함께 풀려고 합니다.
	- 기하 기반(geometry) 탐지의 한계 → BIM 요소의 "존재/완성 여부"는 알지만, 작업 수준(operation-level) 활동(예: 거푸집 vs 콘크리트 타설)은 구분 못 함
	- 외관 기반(appearance) 추론의 적용 범위 한계 → 기존 외관 기반 방법은 이미지가 결합된 점군에만 작동하고, 이미지 없는 레이저 스캔 점군에는 못 씀

2. 사용한 구체적 방법론 (툴/알고리즘)
	[방법론 상세]
	 핵심 성과: BIM 요소 탐지 90%+, 색상 기반 재료 분류 ~68%, 1.4억 점 처리에 ~30초.

3. 서론에 인용하기 좋은 핵심 문장 한 줄
	시공 진척도 모니터링 연구는 as-built 점군과 as-planned BIM 모델을 정합한 뒤 요소 단위로 비교하는 방향으로 발전해 왔으며[ ], ...
```

### Example 2

```markdown
1. 해결하고자 한 핵심 문제 
	1. 실내 시공 현장에서 일반 RGB 이미지만으로 각 BIM 요소의 완성도를 "퍼센트 단위"로 자동 측정하는 것.
	2. 기존 이미지 기반 진척도 모니터링의 두 가지 한계를 정조준합니다.
	  - 이진 분류(완성/미완성)의 한계 → 부분 완성된 요소를 측정 못 함
	  - 퍼센트 평가 방식의 한계 → ① 가림(occlusion), ② 한 프레임에 안 들어오는 대형 요소 문제

2. 사용한 구체적 방법론 (툴/알고리즘)
	[방법론 상세]

3. 서론에 인용하기 좋은 핵심 문장 한 줄
	시공 진척도 모니터링 연구는 as-built 점군과 as-planned BIM 모델을 정합한 뒤 요소 단위로 비교하는 방향으로 발전해 왔으며[ ], 최근에는 합성 데이터로 학습한 인스턴스 세그멘테이션과 SfM 재구성을 결합해 occlusion-free 점군을 생성하는 접근이 제시되었다[ ].
```

## Writing guidelines

- 핵심 문제: 선행 연구 한계를 bullet point로 명시하고, 각 한계에 → 로 해결 방향 연결
- 방법론: 알고리즘/모델 이름은 영어 유지, 설명은 한국어. 성능 수치 반드시 포함
- 인용 문장: 제3자 시점, 학술 문체, 문장 끝에 `[ ]` 인용 자리 포함
